# 看板 v1 — Agent 执行仪表盘

## 核心原则

1. 卡片不由人创建，由 Agent 执行自动涌现
2. ChatPanel 是入口，看板是仪表盘——在对话中说「帮我做X」→ Agent 执行 → 看板自动出现卡片
3. 看板顶部提供统一输入栏「选 Agent → 描述任务 → 执行」——派任务式交互
4. Agent 卡片和手动 sticky note ——两种视觉、两种能力（手动卡片 P2 再做）
5. 点击卡片 = 看到 Agent 完整的执行输出（思考过程/工具调用取决于数据就绪度）
6. 不满意就重跑，不编辑、不评论、不分配

---

## 一、定位：不和 Trello 竞争

Trello 是给人用的项目管理。ai-factory 看板是给「想知道 Agent 在干什么」的人用的。

| | Trello | ai-factory 看板 |
|---|--------|----------------|
| 卡片来源 | 人手创建 | Agent 执行时自动涌现 |
| 状态流转 | 人手拖拽 | Agent 事件自动驱动 |
| 卡片内容 | 人手写的描述 | Agent 的执行输出摘要 |
| 核心操作 | 编辑、评论、分配 | 查看输出、重跑任务 |
| 删卡片 | 归档 | 点 x 就删——这只是一条执行记录 |

---

## 二、两种卡片，两种身份

### Agent 卡片（系统生成）

```
+------------------------------------------+
| :) 每日新闻摘要                          |
| ---------------------------------------- |
| agent: 信息员 . 今天 09:00               |
| 搜索过去 24 小时 AI 行业新闻，           |
| 生成 500 字中文摘要...                   |
| ---------------------------------------- |
| 产出预览：Gemma 3 发布、OpenAI...        |
| 3.2s . (check) 通过                      |
| [查看完整输出]  [重跑]                   |
+------------------------------------------+
```

特征：
- 左侧机器人图标
- 显示来源 Agent 名称
- 显示执行耗时和状态
- 产出预览（前 80 字，点击展开）
- 「重跑」按钮——唯一核心操作
- 不可手写编辑（这是记录，不是草稿）

### 手动卡片（人创建）

```
+--------------------------+
| 确认供应商合同           |
| 附件 x2                  |
| 明天截止                 |
+--------------------------+
```

特征：
- 左侧手写图标
- 可编辑标题和描述
- 删除按钮 hover 出现
- 就是轻量备注卡片

### 视觉区分

| 特征 | Agent 卡片 | 手动卡片 |
|------|-----------|---------|
| 图标 | 机器人 | 手写 |
| 边框 | 左侧 2px 色条 | 普通边框 |
| 来源标识 | agent 名称 | 无 |
| 状态点 | (check)/(x)/(loading) | 无 |
| 底部操作 | 查看输出 . 重跑 | 删除 |

## 三、点击卡片 → 展开 Agent 完整执行过程

当前：卡片只是一个标题。改了以后：

点击 Agent 卡片 → 右侧滑出执行详情面板：

```
+--------------------------------------------+
|  :) 每日新闻摘要                      [x]  |
|  agent: 信息员 . gpt-4 . 今天 09:00       |
|  ---------------------------------------- |
|                                            |

|  +--------------------------------------+ |
|  | 搜索过去 24 小时全球 AI 行业的...     | |

|                                            |
|  思考过程                                  |
|  +--------------------------------------+ |
|  | 让我搜索今天的 AI 新闻...            | |
|  | -> 调用 web_search("AI news")        | |
|  | -> 获取到 12 条结果                  | |
|  | -> 开始摘要...                       | |
|  +--------------------------------------+ |
|                                            |
|  工具调用 (3)                              |
|  . web_search -> 12 条结果                |
|  . web_fetch x2 -> 2 篇文章               |
|                                            |
|  最终输出                                  |
|  +--------------------------------------+ |
|  | 1. Gemma 3 发布...                   | |

|  | ...                                  | |
|  +--------------------------------------+ |

|  3.2s . 2,340 tokens                      |
|                                            |

```

这个面板的内容来自 Agent 执行记录（MemoryStore 里的执行历史）。

## 四、状态自动流转

当前：卡片创建时固定在一个列表。改后：

| Agent 事件 | 卡片动作 |
|-----------|---------|
| agent 开始执行任务 | 卡片出现在「执行中」 |
| agent 执行中 | 状态灯 loading |
| agent 执行完成 | 卡片自动移到「已完成」，状态灯 check |
| agent 执行失败 | 卡片自动移到「需关注」，状态灯 x |
| 定时任务触发 | 同上，但标记来源为「定时任务」 |

用户不拖拽 Agent 卡片。卡片自己会动。拖拽只留给手动卡片。

---

## 五、列结构（v1 三列）

v1 三列就够了。第四列「已暂停」留到 P1——v1 只有 Agent 自动卡片，状态由事件驱动，没有事件会让卡片进入「已暂停」。

| 列 | 含义 | 卡片特征 |
|----|------|---------|
| 执行中 | Agent 正在跑 | loading 脉冲动画 |
| 已完成 | 成功执行 | check 产出预览 |
| 需关注 | 执行失败或产出需人工确认 | x 或 warning |

不叫「To Do」，叫「执行中」——去掉项目管理味道，换成执行监控语言。

## 六、交互规则

| 操作 | Agent 卡片 | 手动卡片 |
|------|-----------|---------|
| 点击 | 打开执行详情面板 | 打开编辑 |
| 拖拽 | 不允许（状态由 Agent 事件驱动） | 允许 |
| 删除 | 长按或右键 → 删除 | 点 x |
| 重跑 | 创建新卡片（保留旧记录），新卡片进「执行中」 | 无 |

## 十二、多 Agent 项目场景

一个工作区可能有多个 Agent——分析员、写手、审查员、代码员。看板是它们的统一仪表盘。

### 新 Agent 自动接入

用户在「我的助手」里新创建一个 Agent → 该 Agent 执行任何任务 → 卡片自动出现在该项目看板上。零配置。

### 按 Agent 筛选

顶部筛选器除了按来源，还支持按 Agent 过滤：

```
[全部 Agent v]  [定时任务 v]
  . 全部 (16)
  . 信息员 (8)
  . 审查员 (4)
  . 代码员 (3)
  . 写手 (1)
```

### Agent 卡片上的来源标识

每张 Agent 卡片显示来源 Agent 名称和色标：

不同 Agent 不同颜色标识，一眼分辨。

### 按 Agent 分列（可选视图）

提供切换开关：

- 默认视图：按状态分列（执行中 / 已完成 / 需关注 / 已暂停）
- Agent 视图：按 Agent 分列（每个 Agent 一列，纵向是状态流转）

Agent 视图适合「我想看某个 Agent 干了什么」的场景。

## 十三、输入栏：派任务式交互（替代「新建卡片」按钮）

v1 不做 [+ 新建卡片] 按钮。看板顶部的统一输入栏：

```
[选 Agent: 信息员 v] [描述要做什么...                    ] [执行 ->]
```

用户选 Agent → 描述任务 → 点执行 → 卡片立刻出现在「执行中」列，Agent 开始跑。

这和 ChatPanel 的区别：
- ChatPanel = 对话式交互，追问、澄清、探索
- 看板输入栏 = 派任务式交互，明确指令，一步执行

同一次执行，在 ChatPanel 和看板都能看到。ChatPanel 是入口（「帮我分析竞品」），看板是仪表盘（看到执行进度和结果）。

手动 sticky note 放到 P2——v1 只做 Agent 卡片。

## 十五、ChatPanel ↔ 看板的方向

> ChatPanel 是入口，看板是仪表盘。

用户在 ChatPanel 说「帮我分析竞品」→ Agent 开始执行 → 看板自动出现卡片。用户切换到看板看进度和结果。

看板卡片底部保留「在对话中查看」入口——点击跳回 ChatPanel 对应的上下文。两个面板双向打通。

## 十六、数据就绪度调查（动工前必须先确认）

### 调查项

| 问题 | 调查方法 |
|------|---------|
| Agent 执行是否有思考过程/工具调用记录？ | 查 MemoryStore 和 runner 的执行记录结构 |
| KanbanSync.on_task_event() 是否已被 Agent runner 调用？ | 查 runner.py 对 kanban_sync 的引用 |
| 单次执行产出是否有结构化存储？ | 查执行输出是否包含完整 prompt、工具调用、最终输出 |

### 根据调查结果调整实施

| 情况 | 策略 |
|------|------|
| 有完整 trace | P0 执行详情面板直接展示思考过程 + 工具调用 |
| 只有输入/输出 | P0 展示 prompt + 输出摘要 + 耗时；思考过程降为 P1 |
| KanbanSync 未接入 | P0 先接入 runner，事件才能驱动状态流转 |

动工前第一步：打开 runner.py 和 memory store，确认这三项。不做假设。
